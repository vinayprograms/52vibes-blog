# TECHNICAL REQUIREMENTS

This document specifies technical requirements for implementing the 52vibes Blog Platform. Requirements are derived from product requirements in `2.1_REQUIREMENTS.md` and incorporate architectural decisions, implementation specifications, and integration patterns.

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

---

## PRODUCTS

1. **Blog Platform** ‚Äî A Hugo-based static website with a custom-built theme implementing tmux-inspired design, Gruvbox theming, and full accessibility support. Deployed to Cloudflare Pages.

2. **crush-export** ‚Äî A Go command-line tool that extracts AI agent session data from the Crush SQLite database and generates structured Markdown documents for GitHub Wiki publication.

3. **Session Log Sync** ‚Äî A GitHub Actions workflow that synchronizes session log Markdown files to the GitHub Wiki, with automated credential health monitoring.

4. **Infrastructure** ‚Äî Cloudflare Pages hosting configuration, DNS settings, security headers, and analytics integration.

### Blog Platform

The Blog Platform is a static website built with Hugo, implementing a custom theme with tmux-inspired design aesthetics, Gruvbox color palette, and comprehensive accessibility features.

#### SPECIFICATIONS

##### Build System

* **BLOG001** - The site SHALL be built using Hugo static site generator version 0.139.0 or later (extended edition for asset processing).
* **BLOG002** - The Hugo version MUST be pinned in `.hugo-version` file at repository root for reproducible builds.
* **BLOG003** - The build MUST NOT require Node.js, NPM, or any JavaScript package manager.
* **BLOG004** - All theme assets MUST be self-contained within the repository; no external theme dependencies SHALL be fetched at build time.
* **BLOG005** - The build command SHALL be `hugo --minify` for production builds.
* **BLOG006** - The output directory SHALL be `public/` (Hugo default).
* **BLOG007** - Hugo Pipes SHALL be used for CSS processing (minification, fingerprinting) without external preprocessors.

##### Theme Architecture

* **BLOG008** - The theme SHALL be implemented as a project-level theme (layouts in root `layouts/` directory), not as a separate theme module.
* **BLOG009** - Template partials SHALL be organized by function in `layouts/partials/` without nested subdirectories for simplicity.
* **BLOG010** - Base template (`baseof.html`) SHALL define blocks: `head`, `header`, `main`, `footer`, `scripts`.
* **BLOG011** - All templates MUST produce valid HTML5 output.
* **BLOG012** - Templates MUST NOT embed inline JavaScript except for the theme toggle script (with CSP hash).

##### CSS Architecture

* **BLOG013** - Styles SHALL be authored in vanilla CSS in `assets/css/main.css`.
* **BLOG014** - CSS SHALL use BEM naming convention for component classes (Block__Element--Modifier).
* **BLOG015** - CSS custom properties (variables) SHALL define the color palette for theme switching.
* **BLOG016** - The CSS file MUST define two theme scopes: `:root` (light default) and `[data-theme="dark"]` (dark override).
* **BLOG017** - Critical CSS for above-the-fold content SHOULD be inlined in `<head>` using Hugo's `resources.Inline` or template inclusion.
* **BLOG018** - The main stylesheet SHALL be loaded with `rel="stylesheet"` after critical CSS, fingerprinted for cache-busting.
* **BLOG019** - Total CSS size SHOULD NOT exceed 30KB uncompressed.

##### Color Palette (Gruvbox)

* **BLOG020** - Dark theme background MUST be `#1d2021` (Gruvbox dark0_hard).
* **BLOG021** - Dark theme foreground MUST be `#ebdbb2` (Gruvbox light1).
* **BLOG022** - Light theme background MUST be `#fbf1c7` (Gruvbox light0).
* **BLOG023** - Light theme foreground MUST be `#3c3836` (Gruvbox dark1).
* **BLOG024** - Accent color MUST be `#d65d0e` (Gruvbox orange) for links and highlights.
* **BLOG025** - Secondary accent MUST be `#458588` (Gruvbox blue) for visited links and secondary elements.
* **BLOG026** - Border color MUST be `#928374` (Gruvbox gray) for tmux-style frames.
* **BLOG027** - All color combinations MUST achieve WCAG 2.1 AA contrast ratio (4.5:1 for body text).

##### Typography

* **BLOG028** - IBM Plex Sans font files MUST be self-hosted in `static/fonts/` directory.
* **BLOG029** - JetBrains Mono font files MUST be self-hosted in `static/fonts/` directory.
* **BLOG030** - Font files MUST be in WOFF2 format only (modern browsers, best compression).
* **BLOG031** - Font-face declarations MUST use `font-display: swap` for text visibility during load.
* **BLOG032** - Font fallback stack for body text SHALL be: `'IBM Plex Sans', -apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif`.
* **BLOG033** - Font fallback stack for code SHALL be: `'JetBrains Mono', 'SF Mono', 'Fira Code', Consolas, monospace`.
* **BLOG034** - Base font size MUST be 18px for optimal readability.
* **BLOG035** - Line height for body text MUST be 1.6.
* **BLOG036** - An Attributions page at `/attributions/` MUST credit IBM Plex Sans (OFL) and JetBrains Mono (OFL) with links to original sources.

##### Layout ‚Äî Tmux Frame

* **BLOG037** - The page layout SHALL implement a tmux-inspired frame with top status bar, content area, and bottom navigation bar.
* **BLOG038** - The frame MUST be implemented using CSS `border` properties with `border-style: solid` and `border-width: 1px`.
* **BLOG039** - ASCII box-drawing characters (‚îÄ‚îÇ‚îå‚îê‚îî‚îò‚îú‚î§‚î¨‚î¥‚îº) MUST NOT be used for structural layout elements.
* **BLOG040** - The content area MUST have `max-width: 800px` for optimal reading line length.
* **BLOG041** - The content area MUST be horizontally centered with `margin: 0 auto`.
* **BLOG042** - On viewports less than 768px, the frame border MAY be removed or simplified for mobile usability.

##### Layout ‚Äî Responsive Breakpoints

* **BLOG043** - Three breakpoint tiers SHALL be implemented: mobile (less than 768px), tablet (768px‚Äì1023px), desktop (‚â• 1024px).
* **BLOG044** - The layout MUST be fluid within each breakpoint tier using relative units (%, rem, vw).
* **BLOG045** - The minimum supported viewport width MUST be 320px.
* **BLOG046** - The maximum content width MUST remain readable at 2560px viewport (no horizontal stretching).
* **BLOG047** - Navigation elements MUST remain accessible at all viewport sizes; hamburger menus MAY be used on mobile if necessary.

##### Header and Branding

* **BLOG048** - The site header MUST display "52 VIBES" branding text prominently.
* **BLOG049** - The branding MUST appear outside/above the tmux-style frame, not contained within bordered areas.

* **BLOG052** - The header SHOULD include a link to the site root (`/`).

##### Top Status Bar

* **BLOG053** - The top status bar SHALL display clickable week numbers for all published weeks.
* **BLOG054** - Week numbers MUST be rendered from `data/weeks.yaml` which contains week metadata (number, status, title, slug).
* **BLOG055** - Only weeks with `status: published` SHALL appear in the status bar.
* **BLOG056** - Each week number MUST be an `<a>` element linking to `/blog/week-XX-<slug>/`.
* **BLOG057** - Each week number MUST have a `title` attribute containing the post title for native tooltip on hover.
* **BLOG058** - The status bar MUST include a "Weeks" label as an `<a>` element linking to `/blog/weeks/`.
* **BLOG059** - Week numbers SHOULD be displayed in a compact format (e.g., "01 02 03 ...") with appropriate spacing.
* **BLOG060** - The current page's week number (if applicable) SHOULD be visually highlighted (e.g., bold, underline, or background).

##### Bottom Navigation Bar

* **BLOG061** - The bottom navigation bar MUST appear on all pages using the `nav-bottom.html` partial.
* **BLOG062** - The navigation MUST include links to: About (`/about/`), Weeks (`/blog/weeks/`), Reports (`/blog/reports/`), RSS (`/blog/index.xml`), GitHub (external repository URL).
* **BLOG063** - All navigation links MUST be `<a>` elements with descriptive text labels.
* **BLOG064** - Icons MAY accompany text labels but MUST NOT replace them; icons SHALL use inline SVG or CSS-based symbols.
* **BLOG065** - The navigation MUST use `<nav>` semantic element with `aria-label="Site navigation"`.
* **BLOG066** - External links (GitHub) MUST include `rel="noopener noreferrer"` and `target="_blank"`.

##### Theme Toggle

* **BLOG067** - A theme toggle control MUST be visible in the header or status bar area.
* **BLOG068** - The toggle MUST switch between light and dark themes by setting `data-theme` attribute on `<html>` element.
* **BLOG069** - Theme preference MUST persist across page loads using `localStorage` with key `theme`.
* **BLOG070** - On initial load, the theme MUST be determined by: (1) localStorage value if set, (2) `prefers-color-scheme` media query otherwise.
* **BLOG071** - The toggle script MUST be inline in the `<head>` (before body rendering) to prevent flash of wrong theme.
* **BLOG072** - The inline script MUST be secured with a CSP SHA-256 hash in the `_headers` file.
* **BLOG073** - The toggle MUST be implemented as a `<button>` element with `aria-label` describing current state.
* **BLOG074** - In CLI browsers or with JavaScript disabled, the page MUST render using system preference or light theme default.

##### Skip Link

* **BLOG075** - A "Skip to content" link MUST be the first focusable element in the DOM.
* **BLOG076** - The skip link MUST be visually hidden by default and become visible on focus (`:focus` styles).
* **BLOG077** - The skip link MUST target the `<main>` element via `href="#main-content"`.
* **BLOG078** - The `<main>` element MUST have `id="main-content"` and `tabindex="-1"` for programmatic focus.

##### Table of Contents

* **BLOG079** - Each blog post MUST display a table of contents (TOC) at the top of the article content.
* **BLOG080** - The TOC SHALL be generated using Hugo's `.TableOfContents` function.
* **BLOG081** - The TOC MUST list all `<h2>` and `<h3>` headings within the post.
* **BLOG082** - Each TOC entry MUST link to the corresponding heading anchor (auto-generated by Hugo).
* **BLOG083** - The TOC MUST be contained in a `<nav>` element with `aria-label="Table of contents"`.
* **BLOG084** - The TOC SHOULD be collapsible on mobile viewports using CSS-only `<details>`/`<summary>` elements.

##### Sticky Section Indicator

* **BLOG085** - Blog posts SHOULD implement a sticky section indicator showing the current section being read.
* **BLOG086** - The indicator MUST be implemented using CSS-only techniques (e.g., `position: sticky` with scroll-snap or intersection observer alternative).
* **BLOG087** - If CSS-only implementation is not feasible, the indicator MAY be omitted rather than requiring JavaScript.
* **BLOG088** - The indicator, if implemented, MUST display format "¬ß Reading: [Section Name]".

* **BLOG090** - Navigation links MUST be generated using Hugo's `.PrevInSection` and `.NextInSection` functions.
* **BLOG091** - Week 1 post MUST NOT display a previous week link.
* **BLOG092** - The most recent published week MUST NOT display an active next week link.
* **BLOG093** - Navigation links MUST display the target post's title, not just "Previous" / "Next".

### crush-export

The crush-export tool is a Go command-line application that extracts session data from the Crush AI agent SQLite database and generates structured Markdown documents suitable for GitHub Wiki publication.

#### SPECIFICATIONS

##### Build and Dependencies

* **CREX001** - The tool SHALL be implemented in Go version 1.23 or later.
* **CREX002** - The Go version MUST be specified in `go.mod` file.
* **CREX003** - SQLite access MUST use `modernc.org/sqlite` (pure Go, no CGO) for cross-platform builds.
* **CREX004** - The tool MUST be buildable with `go build ./cmd/crush-export` without external dependencies.
* **CREX005** - The compiled binary MUST be statically linked and portable.

##### Command-Line Interface

* **CREX006** - The tool MUST support the following flags:
  + `-db <path>` ‚Äî Path to crush.db SQLite database (default: `.crush/crush.db`)
  + `-out <dir>` ‚Äî Output directory for markdown files (default: `sessions`)
  + `-session <id>` ‚Äî Export specific session by ID (optional)
  + `-l` ‚Äî List all sessions without exporting
  + `-h` ‚Äî Display help message
* **CREX007** - With no flags, the tool MUST export all sessions to the default output directory.
* **CREX008** - The tool MUST exit with code 0 on success, non-zero on error.
* **CREX009** - Error messages MUST be written to stderr with descriptive context.

##### Database Schema (Reverse-Engineered from Crush)

* **CREX010** - The tool MUST read from the Crush SQLite database schema (to be reverse-engineered from `github.com/charmbracelet/crush` source).
* **CREX011** - Expected tables (to be verified): `sessions`, `messages`, `files`, or similar.
* **CREX012** - The tool MUST handle schema variations gracefully with informative errors.

##### Session Listing

* **CREX013** - The `-l` flag MUST display a table of sessions with columns: ID, Date, Duration, Messages, Title (or first line of conversation).
* **CREX014** - Sessions MUST be listed in reverse chronological order (newest first).
* **CREX015** - Output MUST be formatted for terminal display with aligned columns.

##### Export Output Structure

* **CREX016** - Each exported session MUST produce one Markdown file.
* **CREX017** - Output filename MUST follow pattern: `YYYY-MM-DD_HH-MM-SS_<sanitized-title>.md`.
* **CREX018** - Title sanitization MUST replace spaces with hyphens and remove special characters.
* **CREX019** - Filename length MUST NOT exceed 100 characters (truncate title if necessary).

##### Markdown Document Structure

* **CREX020** - Each document MUST include the following sections in order:

### Session Log Sync

Session Log Sync is a GitHub Actions workflow that publishes session log Markdown files to the GitHub Wiki, with integrated credential health monitoring.

#### SPECIFICATIONS

##### Wiki Synchronization Workflow

* **SYNC001** - The workflow file MUST be located at `.github/workflows/wiki-sync.yml`.
* **SYNC002** - The workflow MUST trigger on push to `main` branch when files in `sessions/` directory change.
* **SYNC003** - Path filter MUST be: `paths: ['sessions/**']`.
* **SYNC004** - The workflow MUST NOT trigger on pull request events (only merged commits).

##### Authentication

* **SYNC005** - Wiki write access MUST use a fine-grained Personal Access Token (PAT).
* **SYNC006** - The PAT MUST be scoped to: Contents (read/write) for the wiki repository only.
* **SYNC007** - The PAT MUST be stored as repository secret named `WIKI_PAT`.
* **SYNC008** - PAT expiration MUST be set to 90 days maximum.
* **SYNC009** - PAT rotation schedule MUST be documented in repository README.

##### Workflow Steps

* **SYNC010** - Step 1: Checkout main repository with `actions/checkout@v4`.
* **SYNC011** - Step 2: Checkout wiki repository to `wiki/` directory using:
  ```yaml
  git clone https://x-access-token:${{ secrets.WIKI_PAT }}@github.com/<owner>/<repo>.wiki.git wiki
  ```
* **SYNC012** - Step 3: Copy changed session files from `sessions/` to `wiki/`.
* **SYNC013** - Step 4: Commit and push changes to wiki repository.
* **SYNC014** - The workflow MUST only update files that have changed (check `git status`).
* **SYNC015** - The workflow MUST NOT create new wiki pages for files that don't already exist in wiki (prevent zombie pages).
* **SYNC016** - Commit message MUST be: "Sync session logs from main repository".

##### Workflow Output

* **SYNC017** - The workflow MUST output summary of files synchronized.
* **SYNC018** - The workflow MUST fail gracefully if no files need synchronization.
* **SYNC019** - The workflow MUST report authentication failures clearly.

##### Credential Health Monitoring

* **SYNC020** - A second workflow file MUST exist at `.github/workflows/credential-health.yml`.
* **SYNC021** - The health check workflow MUST run on schedule: weekly (every Monday at 09:00 UTC).
* **SYNC022** - The workflow MUST also trigger manually via `workflow_dispatch`.

##### Health Check Steps

* **SYNC023** - Step 1: Attempt to authenticate to GitHub API using `WIKI_PAT`.
* **SYNC024** - Step 2: Query PAT expiration date via GitHub API (if accessible) or attempt a wiki operation.
* **SYNC025** - Step 3: Calculate days until expiration.
* **SYNC026** - Step 4: If expiration less than 14 days, create a GitHub Issue with title "‚ö†Ô∏è WIKI_PAT expires in N days".
* **SYNC027** - The issue body MUST include:
  + Current expiration date
  + Link to GitHub PAT settings page
  + Instructions for rotation
  + Command to update repository secret
* **SYNC028** - The workflow MUST NOT create duplicate issues; check for existing open issue with same title prefix.
* **SYNC029** - If PAT is already expired or invalid, create issue with title "üö® WIKI_PAT is expired or invalid".

##### PAT Rotation Automation

* **SYNC030** - The health check issue MUST include a checklist:
  ```markdown
  ## PAT Rotation Checklist
  - [ ] Generate new fine-grained PAT with wiki scope
  - [ ] Set expiration to 90 days
  - [ ] Update `WIKI_PAT` secret in repository settings
  - [ ] Verify wiki sync by pushing a test change
  - [ ] Close this issue
  ```
* **SYNC031** - The repository README MUST document the PAT rotation process.
* **SYNC032** - A calendar reminder mechanism SHOULD be suggested (e.g., recurring GitHub issue, external calendar).
