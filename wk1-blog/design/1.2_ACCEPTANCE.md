# ACCEPTANCE CRITERIA

This document specifies acceptance criteria for validating that the 52vibes blog platform meets all customer needs. Each criterion maps directly to a need in `1.1_NEEDS.md` and describes how an acceptance testing agent or customer can verify the need has been satisfied.

---

## SPECIFICATIONS

### Core Platform

* **A001** - Verify the blog is accessible at `https://52vibes.dev/blog`.
    * Navigate to the URL in a browser
    * Confirm the blog index page loads without errors
    * Confirm the page displays blog content (not an error page or redirect)

* **A002** - Verify core functionality works without JavaScript.
    * Disable JavaScript in browser settings
    * Navigate to the blog index, a blog post, and all static pages
    * Confirm all content is readable
    * Confirm all internal navigation links work
    * Confirm the table of contents links work within posts

* **A003** - Verify responsive design across device sizes.
    * View the blog on a desktop browser (1920px width)
    * View the blog on a tablet viewport (768px width)
    * View the blog on a mobile viewport (375px width)
    * Confirm layout adapts appropriately at each size
    * Confirm all content remains readable and accessible
    * Confirm navigation remains functional at all sizes

* **A004** - Verify Lighthouse performance score meets threshold.
    * Run Lighthouse audit on the blog index page
    * Run Lighthouse audit on a blog post page
    * Confirm performance score is 95 or higher on both pages

* **A005** - Verify automatic deployment on content update.
    * Make a change to a blog post markdown file
    * Commit and push to the main branch
    * Wait for deployment to complete (observe Cloudflare Pages build)
    * Confirm the change appears on the live site within 5 minutes

* **A006** - Verify CLI browser compatibility.
    * Test with lynx (most restrictive):
        * Open the blog: `lynx https://52vibes.dev/blog`
        * Confirm all text content is readable
        * Confirm navigation links are present and functional
        * Navigate to a blog post and confirm content is readable
        * Confirm table of contents links work
    * Test with w3m (better table support):
        * Open the blog: `w3m https://52vibes.dev/blog`
        * Navigate to Weekly Index page (`/blog/weeks/`)
        * Confirm the 52-week table renders correctly and is readable
        * Confirm table cells are properly aligned and navigable
        * Navigate to a blog post and confirm content is readable

### Visual Design

* **A007** - Verify tmux-inspired visual design.
    * Observe the page layout includes a top status bar
    * Observe the page layout includes a bottom navigation bar
    * Observe the main content area has bordered frame styling
    * Confirm the overall aesthetic resembles a terminal/tmux interface

* **A008** - Verify dark and light theme support.
    * Set system preference to dark mode; confirm blog displays dark theme
    * Set system preference to light mode; confirm blog displays light theme
    * Verify dark theme uses dark background (`#1d2021` or similar) with light text
    * Verify light theme uses light background (`#fbf1c7` or similar) with dark text
    * Verify accent colors are consistent with Gruvbox palette (orange highlights)

* **A009** - Verify theme toggle functionality and graceful degradation.
    * Locate the theme toggle control on the page
    * Click the toggle while in dark mode; confirm switch to light mode
    * Click the toggle while in light mode; confirm switch to dark mode
    * Navigate to another page; confirm theme preference persists (if JS enabled)
    * In CLI browser (lynx), confirm page is still readable without theme toggle functionality

* **A010** - Verify typography.
    * Inspect body text font; confirm IBM Plex Sans is applied
    * Inspect heading fonts; confirm IBM Plex Sans is applied
    * Inspect code blocks; confirm JetBrains Mono is applied
    * Verify fonts render correctly (not falling back to system fonts)

* **A011** - Verify CSS borders are used for terminal styling.
    * Inspect the border elements using browser developer tools
    * Confirm borders are rendered using CSS properties (border, border-radius)
    * Confirm no ASCII box-drawing characters (─, │, ┌, ┐) are used for structural borders

### Navigation Structure

* **A012** - Verify top status bar week navigation.
    * Locate the top status bar
    * Confirm week numbers are displayed for completed weeks only
    * Click on a week number
    * Confirm navigation to the corresponding week's blog post

* **A013** - Verify week number tooltips.
    * Hover over a week number in the status bar
    * Confirm a native browser tooltip appears
    * Confirm the tooltip displays the post title for that week

* **A014** - Verify "Weeks" link to Weekly Index.
    * Locate the "Weeks" label in the status bar
    * Click on "Weeks"
    * Confirm navigation to the Weekly Index page at `/blog/weeks/`
    * In CLI browser, confirm "Weeks" link is accessible and functional

* **A015** - Verify bottom navigation bar links.
    * Locate the bottom navigation bar
    * Confirm links exist for: About, Weekly Index, Reports, RSS, GitHub
    * Click each link and confirm it navigates to the correct destination
        * About → `/about/`
        * Weekly Index → `/blog/weeks/`
        * Reports → `/blog/reports/`
        * RSS → RSS feed URL
        * GitHub → GitHub repository URL

* **A016** - Verify navigation uses text labels.
    * Inspect all navigation elements
    * Confirm each navigation link has descriptive text (not icon-only)
    * If icons are present, confirm they accompany text labels
    * In CLI browser, confirm all navigation is understandable (text visible, no icon-only links)

* **A017** - Verify table of contents in blog posts.
    * Navigate to a blog post with multiple sections
    * Locate the table of contents at the top of the post content
    * Confirm all major sections are listed in the TOC
    * Click a TOC item; confirm the page scrolls to that section

* **A018** - Verify sticky section indicator (if implemented).
    * Navigate to a long blog post
    * Scroll through different sections
    * Observe if a section indicator displays the current section name
    * Confirm the indicator updates as you scroll to different sections
    * Confirm no JavaScript is used for this feature (inspect network/sources)

### Content Pages

* **A019** - Verify About page.
    * Navigate to `/about/`
    * Confirm the page loads without errors
    * Confirm the page contains project description
    * Confirm the page contains author information

* **A020** - Verify Weekly Index page.
    * Navigate to `/blog/weeks/`
    * Confirm a list/table of all 52 weeks is displayed
    * Confirm completed weeks show title and link to post
    * Confirm upcoming weeks are indicated appropriately (no link or different styling)
    * Confirm in-progress week (if any) is distinguishable

* **A021** - Verify Reports section.
    * Navigate to `/blog/reports/`
    * Confirm the reports index page loads
    * Confirm published monthly and quarterly reports are listed
    * Click on a report; confirm it navigates to the full report post

* **A022** - Verify Community page.
    * Navigate to `/community/`
    * Confirm the page loads without errors
    * If community repositories exist, confirm they are displayed
    * If no repositories exist, confirm a friendly message is shown
    * Confirm the page functions with JavaScript enabled

* **A023** - Verify blog index.
    * Navigate to `/blog/`
    * Confirm all published blog posts are listed
    * Confirm each listing shows: title, date, and summary
    * Click on a post title; confirm navigation to the full post

### Blog Posts

* **A024** - Verify weekly post URL structure.
    * Navigate to a weekly blog post
    * Confirm the URL follows the pattern `/blog/week-XX-<title>/`
    * Example: `/blog/week-01-infrastructure/`

* **A025** - Verify monthly report URL structure.
    * Navigate to a monthly report post
    * Confirm the URL follows the pattern `/blog/reports/YYYY-MM/`
    * Example: `/blog/reports/2026-01/`

* **A026** - Verify quarterly report URL structure.
    * Navigate to a quarterly report post
    * Confirm the URL follows the pattern `/blog/reports/YYYY-qN/`
    * Example: `/blog/reports/2026-q1/`

* **A027** - Verify blog post metadata display.
    * Navigate to a blog post
    * Confirm the title is displayed prominently
    * Confirm the publication date is displayed
    * Confirm the estimated reading time is displayed
    * Confirm the quarter/theme indicator is displayed (e.g., "Q1: Agentic Infrastructure")

* **A028** - Verify previous/next navigation.
    * Navigate to a blog post that is not the first or last week
    * Confirm a "Previous week" or similar link exists and works
    * Confirm a "Next week" or similar link exists and works
    * Navigate to Week 1; confirm no "Previous" link
    * Navigate to the latest week; confirm no "Next" link (or disabled)

* **A029** - Verify session log link.
    * Navigate to a blog post
    * Locate the link to the session log
    * Click the link
    * Confirm it opens the corresponding GitHub Wiki page

* **A030** - Verify code/repository link.
    * Navigate to a blog post
    * Locate the link to the week's code or repository
    * Click the link
    * Confirm it opens the correct GitHub repository or directory

### Session Logs

* **A031** - Verify session logs on GitHub Wiki.
    * Navigate to the GitHub Wiki for the blog repository
    * Confirm session log pages exist for completed weeks
    * Confirm content is readable and properly formatted

* **A032** - Verify session log structure.
    * Open a session log wiki page
    * Confirm "Lessons Learned" section exists with content
    * Confirm session statistics are present (duration, message count, etc.)
    * Confirm key conversation excerpts are included (may use collapsible sections)
    * Confirm full conversation log is available (may be in collapsible section)

* **A033** - Verify independent session log updates.
    * Update a session log file in the repository
    * Push the change
    * Confirm the wiki page updates without triggering a blog rebuild
    * Confirm the blog post's link to the session log shows updated content

* **A034** - Verify automatic wiki synchronization.
    * Add or modify a session log markdown file in the `sessions/` directory
    * Commit and push to main branch
    * Observe GitHub Actions workflow runs
    * Confirm the corresponding wiki page is created or updated

* **A035** - Verify crush-export tool processes session data.
    * Confirm `.crush/crush.db` SQLite database exists
    * Run `crush-export -l` to list available sessions
    * Run `crush-export -db .crush/crush.db -out sessions/` to export sessions
    * Confirm the tool produces markdown files in the output directory
    * Confirm output includes the expected structure (Lessons Learned, Stats, Key Exchanges, Full Log)

* **A036** - Verify crush-export tool extracts metadata.
    * Run the crush-export tool on the database
    * Confirm output includes session duration
    * Confirm output includes message count
    * Confirm output includes files created/modified count
    * Confirm output includes error count

### Discovery and SEO

* **A037** - Verify RSS/Atom feed.
    * Navigate to the RSS feed URL (typically `/index.xml` or `/blog/index.xml`)
    * Confirm valid RSS or Atom XML is returned
    * Confirm recent blog posts are included in the feed
    * Test the feed URL in an RSS reader application

* **A038** - Verify meta tags.
    * View page source of a blog post
    * Confirm `<title>` tag contains the post title
    * Confirm `<meta name="description">` tag exists with relevant content
    * Confirm `<meta name="keywords">` tag exists (if applicable)

* **A039** - Verify Open Graph meta tags.
    * View page source of a blog post
    * Confirm `og:title` meta tag exists
    * Confirm `og:description` meta tag exists
    * Confirm `og:image` meta tag exists with social card URL
    * Confirm `og:url` meta tag exists with canonical URL
    * Confirm `og:type` meta tag is set to "article"

* **A040** - Verify Twitter Card meta tags.
    * View page source of a blog post
    * Confirm `twitter:card` meta tag exists (summary_large_image)
    * Confirm `twitter:title` meta tag exists
    * Confirm `twitter:description` meta tag exists
    * Confirm `twitter:image` meta tag exists

* **A041** - Verify sitemap.
    * Navigate to `/sitemap.xml`
    * Confirm valid XML sitemap is returned
    * Confirm sitemap includes blog posts, static pages, and reports
    * Submit sitemap URL to a sitemap validator

* **A042** - Verify JSON-LD structured data.
    * View page source of a blog post
    * Locate `<script type="application/ld+json">` tag
    * Confirm JSON-LD contains Article schema
    * Validate using Google's Rich Results Test or Schema.org validator

* **A043** - Verify social card images.
    * Locate a social card image URL from Open Graph meta tags
    * Open the image URL directly
    * Confirm image displays: week number, post title, quarter theme
    * Confirm image uses consistent template design
    * Test sharing a post URL on social media preview tools (e.g., Twitter Card Validator)

### Engagement

* **A044** - Verify GitHub Discussion link.
    * Navigate to a blog post
    * Locate the link to GitHub Discussions for comments
    * Confirm the link is present and clearly labeled

* **A045** - Verify discussion deep linking.
    * Click the GitHub Discussion link on a blog post
    * Confirm it opens the specific discussion thread for that post
    * Confirm you are not taken to the general discussions page

### Accessibility

* **A046** - Verify semantic HTML structure.
    * View page source or inspect with developer tools
    * Confirm `<header>` element is used for page header
    * Confirm `<nav>` element is used for navigation areas
    * Confirm `<main>` element wraps primary content
    * Confirm `<article>` element wraps blog post content
    * Confirm `<footer>` element is used for page footer

* **A047** - Verify image alt text.
    * Locate any images on blog posts
    * Inspect each image element
    * Confirm all `<img>` tags have non-empty `alt` attributes
    * Confirm alt text is descriptive and meaningful

* **A048** - Verify skip-to-content link.
    * Load a blog page
    * Press Tab key as first action
    * Confirm a "Skip to content" or similar link becomes visible/focused
    * Activate the link; confirm focus moves to main content area

* **A049** - Verify keyboard navigation.
    * Navigate the entire blog using only keyboard (Tab, Enter, Arrow keys)
    * Confirm all interactive elements are reachable
    * Confirm focus indicators are visible on all focused elements
    * Confirm all links and buttons can be activated with Enter/Space

* **A050** - Verify color contrast.
    * Run an accessibility audit tool (Lighthouse, axe, WAVE)
    * Confirm no color contrast violations in dark theme
    * Confirm no color contrast violations in light theme
    * Manually verify text is readable against background colors

* **A051** - Verify logical reading order without CSS.
    * Disable CSS in the browser (or view raw HTML)
    * Read the page content from top to bottom
    * Confirm content appears in a logical, meaningful sequence
    * Confirm no important content is missing or out of order
    * Confirm navigation, main content, and footer appear in expected order

### Content Authoring

* **A052** - Verify Markdown authoring.
    * Examine source files in the content directory
    * Confirm blog posts are stored as `.md` files
    * Confirm standard Markdown syntax is used

* **A053** - Verify code syntax highlighting.
    * Navigate to a blog post containing code blocks
    * Confirm code blocks display with syntax highlighting
    * Confirm highlighting is appropriate for the specified language
    * Confirm highlighting works in both dark and light themes

* **A054** - Verify callout boxes.
    * Create a post with callout/admonition syntax (styled blockquote)
    * Confirm callouts render with distinct styling
    * Verify different callout types are visually distinguishable (tip, warning, note)

* **A055** - Verify local preview capability.
    * Clone the blog repository
    * Run the local preview command
    * Confirm the blog renders locally in a browser
    * Make a change to a post; confirm hot reload or manual refresh shows changes

### Analytics

* **A056** - Verify Cloudflare Analytics.
    * Access Cloudflare dashboard for the domain
    * Navigate to Analytics section
    * Confirm traffic data is being collected
    * Confirm no client-side JavaScript is required for analytics collection

### Community Features

* **A057** - Verify community repository display.
    * Create a public GitHub repository
    * Add the "52vibes" topic to the repository
    * Navigate to `/community/` page
    * Confirm the repository appears in the list (may require cache expiry)

* **A058** - Verify repository information display.
    * On the Community page, examine a listed repository
    * Confirm repository name is displayed and links to GitHub
    * Confirm repository description is displayed (if present)
    * Confirm star count is displayed

* **A059** - Verify API failure handling.
    * Simulate API failure (block GitHub API in network tools, or test offline)
    * Navigate to `/community/` page
    * Confirm a fallback message appears with link to GitHub topics page
    * Confirm page does not show errors or break

* **A060** - Verify empty state message.
    * When no repositories are tagged with "52vibes"
    * Navigate to `/community/` page
    * Confirm a friendly message is displayed (e.g., "No community projects yet")
    * Confirm instructions or encouragement to add the tag is shown

* **A061** - Verify client-side caching.
    * Navigate to `/community/` page; observe repositories load
    * Refresh the page immediately
    * Observe network requests in developer tools
    * Confirm no new API request is made (cached results used)
    * Wait for cache expiry (~5 minutes); refresh; confirm new API request

### Branding

* **A062** - Verify "52 VIBES" branding.
    * Navigate to any page on the blog
    * Confirm "52 VIBES" text is prominently displayed in the header
    * Confirm the branding appears outside/above the tmux-style frame

* **A063** - Verify consistent header.
    * Navigate to multiple pages: blog index, a post, About, Community, Reports
    * Confirm the header with branding appears identically on all pages
    * Confirm header layout and styling is consistent
